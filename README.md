# Nexus AI — Local Development Notes

Quick instructions to get the project running locally and to use auth during development.

Prereqs

- Node 18+ / npm
- PostgreSQL access (Supabase pooler recommended) and `DATABASE_URL` set

Environment

- Create a `.env` with at minimum:

```
DATABASE_URL=postgresql://user:pass@host:5432/dbname
APP_JWT_SECRET=your-dev-jwt-secret
NEXTAUTH_SECRET=your-nextauth-secret
ALLOW_DEV_BOOTSTRAP_CONTEXT=true
```

Dev-only auth helpers

- A dev login route exists at `POST /api/auth/login` that returns a signed bearer token when `ALLOW_DEV_BOOTSTRAP_CONTEXT=true`.
- Example request (JSON body): `{ "email": "owner@nexus.local", "workspaceSlug": "default-workspace" }`
- Use the returned token as `Authorization: Bearer <token>` for API requests that require an actor context.

NextAuth (optional)

- The project includes NextAuth scaffolding. To enable real OAuth, set `GITHUB_ID` and `GITHUB_SECRET` and ensure `NEXTAUTH_SECRET` is set.
- NextAuth session tokens are preferred by the server; `actor-context` will read `getToken()` server-side.

# Nexus AI

Minimal Next.js + Prisma workspace for PDF upload, parsing, and document persistence.

## Quick Start

```bash
npm install
npx prisma generate
npm run dev
```

## Required Environment

Set these in `.env`:

- `DATABASE_URL`: PostgreSQL connection string
- `DEFAULT_APP_USER_EMAIL`: fallback actor email (used for local bootstrap)
- `DEFAULT_WORKSPACE_SLUG`: fallback workspace slug
- `DEFAULT_WORKSPACE_NAME`: fallback workspace display name (optional)
- `MAX_STORED_CONTENT_CHARS`: max characters persisted from extracted PDF text (default `50000`)
- `ALLOW_DEV_BOOTSTRAP_CONTEXT`: set `false` to disable local auto-bootstrap of default user/workspace
- `INTERNAL_AUTH_SECRET`: shared secret for trusted upstream identity headers
- `APP_JWT_SECRET`: secret for signing/validating session JWTs (used by Authorization: Bearer <token>)

## Upload Actor Context

The upload API uses server-side actor resolution in `src/lib/actor-context.ts`.

- In local/dev mode (default), fallback actor/workspace can be auto-bootstrapped.
- In local/dev mode (default), fallback actor/workspace can be auto-bootstrapped.
  In production, auto-bootstrap is disabled unless `ALLOW_DEV_BOOTSTRAP_CONTEXT` is explicitly set; uploads require trusted headers (see below).
- For trusted upstream auth, provide these headers together:
  - `x-internal-auth`: must match `INTERNAL_AUTH_SECRET`
  - `x-user-email`
  - `x-workspace-slug`
- Alternatively, provide a Bearer JWT in the `Authorization` header signed with `APP_JWT_SECRET` and including `email` and `workspaceSlug` claims.

Example (dev-only token mint):

```bash
curl -X POST http://localhost:3000/api/auth/login -H "Content-Type: application/json" -d '{"email":"owner@nexus.local","workspaceSlug":"default-workspace"}'
```

Then call upload:

```bash
curl -X POST http://localhost:3000/api/documents/upload -H "Authorization: Bearer <token>" -F "title=demo" -F "file=@./example.pdf"
```

- Workspace membership is enforced before document creation.

## CI integration

An example GitHub Actions workflow that asserts the upload endpoint rejects unauthenticated requests is included at `.github/workflows/integration-upload-auth.yml`.

The workflow:

- Starts a local Postgres service
- Runs `npx prisma db push` and `npm run build`
- Starts the app and verifies `/api/documents/upload` returns `401` without `x-internal-auth`
- Verifies requests with `x-internal-auth`, `x-user-email`, and `x-workspace-slug` are accepted

Adjust the workflow secrets for your org before enabling it.
